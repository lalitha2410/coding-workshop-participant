# Allocations service — tests

Same structure as the other backend service tests. Allocations is a join table
(resources <-> projects), so tests also cover the relationship-specific rules.

- **`test_validation.py`** — pure validation: required integer `resource_id`/
  `project_id`, `allocation_pct` 0–100, date formats, and the **date-order rule**
  (`end_date` not before `start_date`).
- **`test_handler.py`** — router with the repository monkeypatched. Covers
  routing, status codes (200/201/204/400/404/405/500), the **dual-FK guard**
  (missing resource and/or project → 400), **duplicate pair → 400**, the
  **`/over-allocated`** and **`/summary`** analytics routes, request parsing
  (API Gateway v1 + v2), and JSON serialization.
- **`test_integration.py`** — real-PostgreSQL round-trips plus:
  - unique `(resource_id, project_id)` → `DuplicateAllocationError`,
  - missing FK → `ForeignKeyViolation`,
  - the **over-allocation calculation** (60% + 60% = 120% flagged; 50% not; a
    boundary case that exactly 100% is *not* over-allocated).
  Throwaway parent projects/resources are cascade-deleted on teardown; the
  module auto-skips when the DB is unavailable.

## Run

```sh
pip install -r backend/allocations/requirements-dev.txt

# Unit tests only (no database needed)
pytest backend/allocations/tests/test_validation.py backend/allocations/tests/test_handler.py

# Everything, including integration against local Postgres
IS_LOCAL=true pytest backend/allocations/tests/
```

## Analytics endpoints

| Method | Endpoint                      | Description                                         |
| ------ | ----------------------------- | --------------------------------------------------- |
| GET    | `/allocations/over-allocated` | Resources whose summed allocation exceeds 100%      |
| GET    | `/allocations/summary`        | Per-resource allocation totals with `over_allocated`|

> `postgres_service.py` here is generated by `bin/sync-shared.sh` from
> `backend/_shared/`. Run the sync (or a deploy) before packaging.
